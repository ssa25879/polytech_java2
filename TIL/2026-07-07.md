### Organization 생성

- GitHub 상단에 + 버튼 -> New Oraganization 으로 생성

### Repository 생성

- 상단의 + 버튼을 눌러 New Repository 생성
- 팀원을 Collaborator로 추가
- 모든 인원은 해당 Repo를 Clone

### 프로젝트 생성

- 좌상단 Projects 탭 -> New project -> 원하는 템플릿 선택
- project 이름 설정 후 생성
- project는 기본적으로 private이기 때문에 Setting에서 public으로 설정 필요

## dev 브랜치 생성

- 브랜치 메뉴에서 dev 브랜치를 생성
- Settings 에서 Default branch를 dev로 설정

## 이슈 작성

- Repo 에서 Issues -> New issue
- 이슈 작성은 작은 단위로 진행
- 우측 Assignees 에서 담당자를 지정
- Labels 지정
- Projects 지정
- 생성 완료 후 Projects 탭에서 확인

## 이슈 관리

- 우측 Projects 하위 메뉴 중 Status를 통해 상태 변경 가능
- 우측 Development 하위 메뉴 중 Create a branch를 통해 해당 Repo에 브랜치 생성
- `git fetch origin` `git checkout (브랜치명)`을 통해 브랜치 변경 - 복사 가능 메뉴 있음
- 위 명령어 실행 후 작업 시작 (커밋 - 푸시 - PR 작성)

## PR 작성

- dev 브랜치인지 확인 후 PR 작성
- 올바른 PR 메시지 예시

```markdown
PR 제목 : 회원 가입 시 이메일 중복 체크 기능 추가

### 변경 사항

- 회원 가입 시 사용자가 입력한 이메일이 이미 존재하는지 확인하는 기능을 추가했습니다.
- EmailRepository에 `existsByEmail` 메서드를 추가하여 이메일 중복 여부를 조회할 수 있도록 구현했습니다.
- 회원 가입 서비스에 중복 이메일 확인 로직을 추가하여, 중복 이메일일 경우 예외를 발생시키도록 했습니다.

### 테스트

- 중복 이메일이 입력될 경우 409 Conflict 상태 코드를 반환하는지 확인하는 테스트 케이스를 작성했습니다.
- 중복이 아닌 이메일일 경우 회원 가입이 정상적으로 진행되는지 확인하는 테스트 케이스도 추가했습니다.

### 참고 사항

- 이메일 중복 체크 로직 추가에 따른 API 문서 업데이트 필요
- 중복 이메일 예외 처리 로직에 대한 코드 리뷰 부탁드립니다.
```

- 올바른 PR 메시지에는 코드 변경 사항의 목적, 내용을 명확하게 전달해야함

## Review / Merge

- PR 작성 후 해당 코드 리뷰 후 Merge 진행
- Merge 된 브랜치는 삭제
- 삭제까지 완료한 후 Projects 탭에서 Done으로 이동하였는지 확인
- 로컬에서 dev 최신정보 가져옴 `git checkout dev` `git pull`
- 로컬에 남은 기능 브랜치 삭제 `git branch -d (브랜치명)`
- 완료 후 다시 이슈 생성부터 루틴 시작
